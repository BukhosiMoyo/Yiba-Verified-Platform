# YibaWise / YibaVerified: The National Digital Infrastructure for Compliance, Trust & Skills Development

## SECTION 1: The Big Problem We Are Solving (Context & Urgency)

### 1.1 The Reality on the Ground
Across South Africa, regulatory compliance and learner management are still handled through fragmented, manual processes.
Institutions are expected to:
- Upload CSV files
- Send documents via email
- Prepare paper-based evidence
- Wait for site visits
- Respond to ad-hoc requests with no shared system of record

As a result:
- There is no real-time visibility for regulators like QCTO
- Institutions struggle to stay consistently compliant
- Compliance becomes stressful, reactive, and error-prone
- Learners are exposed to risk, misinformation, and abuse

The system does not fail because people are unwilling — it fails because the tools are outdated.

### 1.2 The QCTO Banner Problem (A Real-World Signal)
At the entrance of the QCTO offices, a prominent warning is displayed:
“Beware of unaccredited institutions.”

The solution offered today is:
“If you encounter this problem, send us an email.”

This highlights a deeper issue:
- Reporting is reactive, not proactive
- Communication is unstructured
- There is no system that prevents the problem before it happens
- There is no shared source of truth

Email is being used to solve a systemic national problem.

### 1.3 Core Insight
Compliance is not failing because of bad intentions.
Institutions want to comply.
Regulators want visibility.
Learners want protection.

Compliance is failing because of bad systems.

When compliance is treated as:
- A once-off submission
- A manual exercise
- A disconnected process

It becomes fragile.

The real opportunity is to embed compliance into daily operations, so that:
- Data is captured once
- Evidence is generated automatically
- Visibility exists at all times
- Trust is built continuously, not after the fact

This platform exists to solve that problem at the system level.

## SECTION 3: The Full Institution Journey (High-Level)
This section describes the end-to-end lifecycle of an institution on the platform.
Not as features. Not as modules. But as a journey — from first contact to long-term value.

### 3.1 The Lifecycle View
Every institution moves through the same high-level stages:

1. **Awareness**
   Initial discovery through targeted outreach and education.

2. **Trust-Building**
   Understanding the problem, seeing alignment with QCTO, and recognising credibility signals.

3. **Soft Onboarding (Review Invitation)**
   A low-pressure, explainer-driven introduction to the platform’s value.

4. **System Onboarding (Institution Setup)**
   Formal entry into the system: profile creation, structure setup, and access control.

5. **Operational Usage (LMIS + Submissions)**
   Day-to-day usage: learners, attendance, documents, and internal processes.

6. **Compliance Visibility (QCTO Interaction)**
   Live oversight, structured submissions, and request-based interaction with regulators.

7. **Reputation & Recognition (Public Trust Layer)**
   Performance signals surface as credibility, not marketing claims.

8. **Ecosystem Participation (Students + Employers)**
   Institutions benefit from student visibility, CV portals, and employer engagement.

9. **Monetisation (Introduced Later)**
   Premium features unlocked only after value is already proven.

### 3.2 Why This Journey Matters
Most systems fail because they:
- Start with forms
- Demand commitment too early

This journey is intentionally designed to:
- Reduce friction at entry
- Build confidence before commitment
- Delay monetisation until value is obvious

Institutions never feel “sold to”.
They feel supported into a better operating reality.

### 3.3 A Compounding System
Each stage naturally unlocks the next:
Awareness feeds trust
Trust enables onboarding
Onboarding enables usage
Usage generates compliance
Compliance builds reputation
Reputation drives growth

This creates a self-reinforcing loop where:
- Institutions want to stay
- Regulators gain clarity
- The system improves with scale

## SECTION 5: Invitations, Onboarding & Access Rules
This section defines who gets access, how they get access, and what experience they receive.
The system is intentionally strict here — because clarity prevents confusion and misuse.

### 5.1 Role-Specific Invitation Logic
Not all users are invited the same way.
Each role has a deliberate entry path:

- **Platform Admin**: Direct access. No onboarding. Full system visibility.
- **QCTO Users**: Guided onboarding. Contextual walkthrough. System purpose explained before use.
- **Institution Admins**: Two-path invitation flow. Review option or direct onboarding.
- **Institution Staff**: Direct, scoped access. No marketing or review flow. Only what they need to perform their role.

This ensures:
No unnecessary friction. No accidental exposure. No role confusion.

### 5.2 Two Distinct Paths for Institutions
Institution Admins are the only role that receives a choice.

**Path 1: Review Invitation**
Used when trust and understanding still need to be built.
Includes:
- Step-by-step walkthrough
- Clear explanation of:
  - What the platform does
  - What data will be required
  - How compliance becomes easier
- Visual, guided experience
- No account creation pressure

This path is educational, not transactional.

**Path 2: Access Dashboard**
Used when the institution is ready to proceed immediately.
Includes:
- Account creation
- Institution setup
- Profile completion
- Operational access
No review. No delays. Straight to value.

### 5.3 Decline Flow (Feedback Loop, Not a Dead End)
Declining an invite is treated as information, not rejection.
If an institution declines:
- They must select one structured reason
- Optional message field is available
- Data is stored against their profile

Examples:
- Using another system
- Manual processes preferred
- Not ready at this time
- Other (with explanation)

This creates:
- Insight into market resistance
- Future re-engagement opportunities
- Continuous improvement feedback
Declines are never ignored — they are analysed.

### 5.4 Why This Matters
This structure ensures:
- Institutions enter with clarity
- Regulators enter with context
- Admins enter with confidence

Most importantly:
No one is forced. No one is confused. No one is overwhelmed.
By the time operational use begins, alignment already exists.

## SECTION 6: Core Operational Layer (LMIS + Compliance)
This is the engine room of the platform.
Everything before this builds trust — everything after this compounds value.
The goal here is simple:
If institutions run their daily operations inside the system, compliance happens automatically.

### 6.1 Qualifications & Programmes (The Backbone)
All operations are anchored to structured qualification data.
Each qualification includes:
- Qualification name
- Qualification code
- NQF level
- Qualification type (Occupational / Skills / Learnership, etc.)
- Duration
- Study mode (Full-time, Part-time, Blended)
- Associated programmes / courses

This data is:
- Created once
- Reused everywhere
- Never duplicated manually

Used across:
Readiness (Form 5) → Attendance registers → Learner enrolment → CV & student profiles → Submissions to QCTO
One source of truth.

### 6.2 Learner Management (Live, Not Retrospective)
Learners are managed as active records, not static entries.
Core features:
- Learner enrolment per qualification
- Class / cohort assignment
- Status tracking (active, completed, withdrawn)
- Linked evidence and documents
- POPIA-aware data handling

Every learner action is:
- Timestamped
- Institution-scoped
- Audit-ready
This ensures QCTO can trust the data without asking for CSVs.

### 6.3 Attendance Register System (Designed for Reality)
Attendance is captured the way institutions actually work.
Key principles:
- Calendar-based view
- Course and class specific
- Default “All Present” logic
- Quick absence marking

If a learner is absent:
- Absence is recorded
- Optional reason captured
- Sick note upload supported (PDF / image)

Registers are:
- Editable only within allowed windows
- Exportable as official documents
- Directly usable for submissions
- Fully auditable

Attendance is no longer something you “prepare later” — it’s something you record once and reuse everywhere.

### 6.4 Why This Layer Is Critical
This layer:
- Eliminates duplicate work
- Removes manual reporting
- Creates real-time compliance visibility
- Protects institutions from accidental non-compliance

Most importantly:
Institutions don’t work “for compliance” — they just work.
The system does the rest.

## SECTION 7: Submissions & QCTO Requests Engine
This section replaces the most painful part of the current system:
Manual exports, CSV files, email attachments, and back-and-forth clarification.

The goal is to make submissions:
Structured. Verifiable. Generated directly from live system data.
No room for manipulation. No room for confusion.

### 7.1 Structured Submissions (No Manual Files)
Institutions do not upload arbitrary files as submissions.
Instead, they:
- Create a new submission
- Select what they are submitting
- The system generates the submission from verified data

Examples of submission types:
- Attendance records
- Learner enrolment lists
- Assessment results
- Qualification readiness evidence
- Policy and compliance documents

Each submission includes:
- Submission name (human-readable)
- Submission type(s)
- Qualification / course context
- Date range (where applicable)
- Auto-generated data references

Submissions are:
- Immutable once submitted
- Versioned if re-submitted
- Fully traceable

### 7.2 Submissions Are Data, Not Documents
Instead of “uploading attendance”:
- The system references attendance already captured
- Displays summaries (e.g. attendance %)
- Allows drill-down to learner-level detail
- QCTO reviewers can view dashboards, inspect underlying data, or download official exports only if needed.

This flips the model:
The system becomes the source of truth — not the attachment.

### 7.3 QCTO Requests (Pull, Not Push)
QCTO can initiate structured requests for information.
Request examples:
- Attendance for Qualification X (last 90 days)
- Learner list for Programme Y
- Updated policy documents
- Evidence for readiness item Z

Each request includes:
- Request type
- Required data scope
- Deadline (e.g. 7 / 14 / 30 days)
- Optional notes

Institutions:
- See requests clearly in their dashboard
- Respond by approving system-generated data
- Upload supporting documents only where required

### 7.4 Deadlines, Status & Accountability
Every request and submission has:
- Status (Pending / Submitted / Reviewed / Returned)
- Deadline tracking
- Automatic reminders
- Audit trail

Both sides can see:
- Who requested
- Who responded
- When it was completed
- What data was provided
No ambiguity. No “we sent it already” disputes.

### 7.5 Why This Engine Changes Everything
This engine:
- Eliminates false reporting
- Reduces regulator workload
- Protects institutions from mistakes
- Builds long-term trust in the data
Most importantly:
Compliance becomes a continuous process — not a stressful event.

## SECTION 11: System Architecture & Scalability
This platform is designed to be infrastructure-agnostic, future-proof, and government-ready from day one.
Not a quick MVP. Not a fragile startup system.
A national-scale foundation.

### 11.1 Modular by Design
The system is built as a modular platform, not a tangled monolith.
Core modules include:
- Identity & Roles
- Institutions
- Learners
- Qualifications
- Attendance
- Submissions
- QCTO Requests
- Reputation Engine
- CV & Employer Portal

Each module has clear boundaries, can evolve independently, and can be extended without breaking others.

### 11.2 Multi-Tenant at the Core
Institutions are logical tenants, not separate apps.
This means:
- Strong data isolation
- Shared infrastructure
- Lower operating cost
- Easier national rollout
Every query is scoped by institution context and role.

### 11.3 Cloud-Native, But Not Cloud-Locked
The platform:
- Runs on modern cloud infrastructure
- Uses standard technologies (Node.js, PostgreSQL)
- Can be containerised (Docker)

It can be deployed to:
- Commercial cloud (AWS, Azure, GCP)
- Government private cloud
- On-premise data centres
No vendor lock-in.

### 11.4 Scales With Usage, Not Assumptions
The system scales:
- Horizontally (more users, more institutions)
- Vertically (more data, more documents)

Heavy assets (documents, images) are separated from core transactional data and reporting logic.
This keeps performance stable even under load.

### 11.5 Audit-First Architecture
Every important action is logged, traceable, and has a responsible user.
This makes:
- Investigations easier
- Disputes resolvable
- Oversight credible
The system assumes audits will happen — and welcomes them.

### 11.6 Designed for Long-Term Ownership
This is not a “black box” system.
The architecture:
- Uses open standards
- Is readable by any competent dev team
- Can be handed over if required
Ownership can change. Trust does not break.
